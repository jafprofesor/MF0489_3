4.2 Teoría: ¿Qué es -L?

┌─────────────┐                    ┌──────────────┐
│  Tu Kali    │                    │ Ubuntu Server│
│             │    SSH Tunnel      │              │
│ localhost   │◄──────────────────►│ 127.0.0.1    │

Comando:

5. Práctica 3: Dynamic Port Forwarding (-D) - SOCKS Proxy

Duración: 60 minutos
Dificultad: ⭐⭐⭐ Intermedia-Avanzada

5.1 Objetivos

  • Configurar un proxy SOCKS5 dinámico
  • Entender la diferencia entre -L y -D
  • Navegar “desde dentro” de la red del servidor

5.2 Teoría: ¿Qué es -D?

┌─────────────┐                    ┌──────────────┐
│  Tu Kali    │                    │ Ubuntu Server│
│             │    SSH Tunnel      │              │
│ localhost   │◄────SOCKS5────────►│   Cualquier  │
│   :9092     │      Proxy         │   destino    │
│             │                    │              │
└─────────────┘                    └──────────────┘
     ↓                                      ↓
 curl --proxy                       192.168.1.50:80
 socks5h://                         10.0.0.5:443
 localhost:9092                     google.com:80

Diferencia clave:

  • -L: Destino fijo y predefinido
  • -D: Destino dinámico, decidido en tiempo de ejecución

5.3 Ejercicios

Ejercicio 3.1: Crear Proxy SOCKS

# Establece el túnel dinámico

# Verifica
ss -tlnp | grep 9092

Ejercicio 3.2: Prueba con curl

Ejercicio 3.5: proxychains (Herramienta Avanzada)

# Instala proxychains (si no está)
sudo apt update && sudo apt install -y proxychains4

# Configura
echo "socks5 127.0.0.1 9092" | sudo tee -a /etc/proxychains4.conf

# Usa cualquier herramienta a través del proxy:

proxychains4 nmap -sT 192.168.1.0/24
proxychains4 ssh studentXX@192.168.1.50

# ⚠️ Nota: nmap puede ser lento vía proxy

5.4 Caso de Uso Real

Escenario: Eres pentester y necesitas acceder a una red interna (10.0.0.0/24) que solo es visible desde el servidor Ubuntu comprometido.

# 1. Crea proxy SOCKS
ssh -D 9092 studentXX@192.168.99.50

# 2. Escanea la red interna
proxychains4 nmap -sT -Pn 10.0.0.0/24

# 3. Accede a servicios descubiertos
proxychains4 curl http://10.0.0.5:80

5.5 Comparativa: -L vs -D

Característica -L (Local) -D (Dynamic)
Destino Fijo, predefinido Dinámico, cualquier IP
Configuración Simple Requiere app con soporte proxy
Casos de uso 1 servicio específico Múltiples servicios/red completa
Overhead Bajo Medio ( SOCKS protocol)
Flexibilidad Baja Alta

6. Práctica 4: Remote Port Forwarding (-R)

Duración: 50 minutos
Dificultad: ⭐⭐⭐ Intermedia-Avanzada

6.1 Objetivos

  • Comprender el concepto de Remote Port Forwarding
  • Exponer servicios locales a un servidor remoto
  • Entender casos de uso en desarrollo y soporte

6.2 Teoría: ¿Qué es -R?

┌─────────────┐                    ┌──────────────┐
│  Tu Kali    │                    │ Ubuntu Server│
│             │    SSH Tunnel      │              │
│ 127.0.0.1   │◄──────────────────►│ localhost    │
│   :3000     │                    │   :9091+XX   │
│             │                    │              │
└─────────────┘                    └──────────────┘
     ↓                                      ↓
 Servicio local                    ssh desde Ubuntu
 (tu máquina)                      curl localhost:9091

Dirección inversa:

  • -L: Local → Remoto (accedes TÚ al servidor)
  • -R: Remoto → Local (el servidor accede a TI)

6.3 Ejercicios

Ejercicio 4.1: Servicio Local de Demo

# Crea un servicio web simple en tu Kali
mkdir -p /tmp/mi_servicio
echo "<h1>Servicio de Alumno XX</h1><p>$(date)</p>" > /tmp/mi_servicio/index.html

cd /tmp/mi_servicio
python3 -m http.server 3000 --bind 127.0.0.1 &

# Verifica
curl http://localhost:3000

Ejercicio 4.2: Túnel Remote Manual

# Establece el túnel -R
# NOTA: El puerto remoto debe ser único por alumno
# student01 → 9091, student02 → 9092, etc.
PUERTO_REMOTO=$((9090 + XX))

ssh -i ~/.ssh/id_rsa_lab \
    -R ${PUERTO_REMOTO}:localhost:3000 \
   -N -f studentXX@192.168.99.50

# Verifica DESDE el servidor Ubuntu:
ssh studentXX@192.168.99.50 "ss -tln | grep $PUERTO_REMOTO"

Ejercicio 4.3: Acceso desde el Servidor

# Conéctate al servidor y prueba el túnel
ssh studentXX@192.168.99.50

# Dentro del servidor:
curl http://localhost:9091  # (cambia 9091 por tu puerto)

# Deberías ver el HTML de tu servicio local en Kali

# Sal
exit

6.5 Comparativa: -L vs -R

Característica -L (Local) -R (Remote)
Dirección Tú → Servidor Servidor → Tú
Quién inicia Cliente SSH Cliente SSH

Puerto escucha Tu máquina Servidor remoto
Caso típico Acceder a BD interna Demo desarrollo

Ejercicio 5.5: Multiplexación SSH (Avanzado)

# En lugar de 3 conexiones SSH, usa 1 con multiplexación

# 1. Crea directorio de control
mkdir -p ~/.ssh/sockets

# 2. Añade a ~/.ssh/config:
cat >> ~/.ssh/config << 'EOF'
Host ubuntu-server
   HostName 192.168.99.50
    User studentXX
    ControlMaster auto
    ControlPath ~/.ssh/sockets/%r@%h-%p
    ControlPersist 600
EOF

# 3. Primera conexión (crea socket maestro)
ssh -i ~/.ssh/id_rsa_lab ubuntu-server

# 4. Los siguientes túneles reusan la misma conexión:
ssh -L 9090:localhost:8080 -N -f ubuntu-server
ssh -D 9092 -N -f ubuntu-server
ssh -R 9091:localhost:3000 -N -f ubuntu-server

# Verifica: solo 1 proceso SSH maestro
ps aux | grep ssh

8.2 Escenario

Contexto:
El servidor Ubuntu (192.168.99.50) tiene acceso a una red interna “oculta” (10.10.10.0/24) que no es reachable directamente desde tu Kali. Dentro de esa red hay:

  • Un servidor web (10.10.10.50:80)
  • Una base de datos MySQL (10.10.10.60:3306)
  • Un servidor SSH (10.10.10.70:22)

Objetivo: Acceder a estos 3 servicios desde tu Kali.

8.3 Ejercicios

Ejercicio 6.1: Reconocimiento

# 1. Conéctate al servidor Ubuntu
ssh studentXX@192.168.99.50

# 2. Identifica interfaces de red
ip addr show

# 3. ¿Tiene acceso a 10.10.10.0/24?
ping -c 2 10.10.10.50

# 4. Escanea la red interna (desde Ubuntu)
sudo nmap -sT -Pn 10.10.10.0/24

# 5. Sal
exit

Ejercicio 6.2: Túnel -L a Servidor Web

# Crea túnel al web interno
ssh -i ~/.ssh/id_rsa_lab \
    -L 8080:10.10.10.50:80 \
   -N -f studentXX@192.168.99.50

# Accede desde tu Kali
curl http://localhost:8080

# O en navegador:
firefox http://localhost:8080 &

Ejercicio 6.3: Túnel a MySQL

# Crea túnel a BD
ssh -i ~/.ssh/id_rsa_lab \
    -L 3307:10.10.10.60:3306 \
   -N -f studentXX@192.168.99.50

# Conecta (si tienes mysql-client)
mysql -h 127.0.0.1 -P 3307 -u root -p

# O verifica con nmap
nmap -sV -p 3307 localhost

Ejercicio 6.4: Proxy SOCKS para Acceso Total

# En lugar de túneles individuales, usa -D
ssh -i ~/.ssh/id_rsa_lab \
    -D 9092 \
   -N -f studentXX@192.168.99.50

# Configura proxychains
echo "socks5 127.0.0.1 9092" | sudo tee -a /etc/proxychains4.conf

# Escanea la red interna DESDE tu Kali
proxychains4 nmap -sT -Pn 10.10.10.0/24

# Accede a cualquier servicio:
proxychains4 curl http://10.10.10.50:80
proxychains4 ssh root@10.10.10.70

Ejercicio 6.5: Túnel en Cascada (Avanzado)

# Si 10.10.10.70 tiene SSH, crea un túnel en cascada:

# 1. Primero al servidor Ubuntu
ssh -i ~/.ssh/id_rsa_lab \
    -L 2222:10.10.10.70:22 \
   -N -f studentXX@192.168.99.50

# 2. Ahora conecta al SSH interno
ssh -p 2222 root@localhost

# ¡Estás en el servidor interno!

Ejercicio 7.1: Auditoría de Túneles Propios

# Lista todos tus túneles SSH

Ejercicio 7.2: Puertos Escuchando

# Verifica qué puertos están abiertos en tu Kali

# ¿Cuáles son de túneles SSH?
ss -tlnp | grep ssh

Anexo A: Comandos de Referencia

A.1 Comandos Esenciales

# Local Port Forwarding (-L)
ssh -L [puerto_local]:[host_remoto]:[puerto_remoto] usuario@servidor

# Dynamic Port Forwarding (-D)
ssh -D [puerto_local] usuario@servidor

# Remote Port Forwarding (-R)
ssh -R [puerto_remoto]:localhost:[puerto_local] usuario@servidor

# Combinar opciones comunes
ssh -i ~/.ssh/id_rsa -o StrictHostKeyChecking=no -N -f usuario@servidor

# Ver túneles activos
ps aux | grep "ssh.*-[LDR]"
ss -tlnp | grep ssh

# Cerrar túnel específico
pkill -f "ssh.*-L.*9090"

# Ver logs de SSH
tail -f /var/log/auth.log  # En servidor
journalctl -f -u ssh       # En systemd
